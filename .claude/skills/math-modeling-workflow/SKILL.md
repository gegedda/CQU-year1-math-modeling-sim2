# 数学建模竞赛 AI 协作工作流

## 触发条件

当对话中出现以下任一信号，加载本 Skill：

- "数学建模" / "数模" / "美赛" / "国赛" + "帮我"
- "选哪道题" / "分析赛题" / "制定计划" / "三天怎么安排"
- "搭数模项目" / "生成论文模板" / "整理参考文献"
- "按数模工作流" / "数模项目结构"

## 核心理念

> AI = 代码队友 + 文献助手 + 图表工具。数学模型建立、关键假设提出、结果解译 → 队员主导。

## 工作流（7 阶段）

### 阶段 0 · 赛前准备（Day 0）

```
1. Python 3.8+ / numpy / scipy / matplotlib / pandas
2. 建立项目目录（见附录 A）
3. README.md — 赛事信息 + 提交物清单 + AI 红线
4. .gitignore（Python 标准项）+ .gitattributes（行尾规范）
5. git init + GitHub 远程仓库
```

**→ 提示词：** "我要参加[比赛]，选题[A/B/C]，截止[日期]。先搭建项目结构、写 README、初始化 git。"

---

### 阶段 1 · 选题分析（Day 1 上午）

**决策矩阵：**

| 维度 | 权重 | 含义 |
|------|:--:|------|
| AI 可辅助程度 | 5 | 代码/公式量越大越有优势 |
| 知识门槛 | 4 | 题目所需 vs 已有知识 |
| 数据可得性 | 3 | 赛题提供 > 自己找 |
| 创新空间 | 3 | 是否有明显改进切入点 |

**→ 提示词：** "这是我的赛题文件。我们队大一大二，知识有限。分析每道题对 AI 辅助的友好程度，用决策矩阵推荐选题。"

**实战教训：** A 题（力学）→ 知识断层大 · **B 题（优化类）→ 最优** · C 题（深度学习）→ 需 GPU

---

### 阶段 2 · 任务规划（Day 1 上午）

**3 人队典型分工：**
- 代码主力：Day 1 全天 → Day 2 可不在 → Day 3 回归复查
- 数据队友：Day 1 全程 → Day 2 独立推进
- 论文手：Day 1-2 旁观 → Day 3 全天写作

**铁律：** Day 1 完成所有核心代码，Day 2 只做验证补全。每项任务标注预计耗时和产出。

**→ 提示词：** "我们有3天。我的情况是[说明各成员时间]。制定三日计划：(1)Day 1 逐时段任务 (2)Day 2 独立操作手册（傻瓜式，复制粘贴即可）(3)Day 3 论文手流程。标注耗时。"

---

### 阶段 3 · 基础设施 + 核心求解（Day 1 下午-晚间）

**共用模块：**
```
src/utils/
├── data_loader.py      # CSV → DataFrame
├── distance.py          # 欧氏/曼哈顿距离矩阵
└── visualization.py     # matplotlib 统一风格 + 中文
```

**每问题目录：**
```
src/problem1/
├── model.md             # 先写！符号表+假设+定理+伪代码
├── solve_all.py         # 至少 3 种算法，可独立运行
├── sensitivity_analysis.py
└── algorithm_comparison.py
```

**代码规范：**
1. 每脚本可独立运行（`python xxx.py`）
2. 文件头注释：用途、输入、输出
3. `snake_case` 命名 · `np.random.seed(42)` · 单文件 ≤300 行
4. **禁止硬编码路径** — `os.path.dirname(__file__)` / `%~dp0`

**→ 提示词：** "先读数据文件理解格式。写共用模块。再逐个问题：先写 model.md，再写 .py 代码。"

**⚠️ 常见事故：**
- AI 把速度 100 mm/s 写成 600 mm/s → **每个参数对照赛题原文验证**
- AI 把随机初始分配当硬约束 → **人类必须先指出数据性质**
- 只实现一个算法 → **至少 3 种 + 小规模精确解验证**

---

### 阶段 4 · 深度分析（Day 1 晚间 + Day 2）

**4.1 参数敏感性：** 每个关键参数 ≥5 个取值点全量扫描 → 找出哪个参数最关键

**4.2 文献驱动改进（高价值）：** 精读 15-20 篇 → 提取可实现思路 → AI 实现 → 对比效果 → **负结果也记录**

**4.3 多维度对比矩阵：** 算法 × 规模 = 全量表 + 图 · 策略 × 数据 = 横向对比

**→ 提示词：** "基于精读文献，提取 3-5 个可实现的改进思路。每思路实现代码、跑实验、对比。负结果也输出分析。"

---

### 阶段 5 · 论文支撑（Day 2-3）

**5.1 每问题一个 `model.md`：**
```
一、符号说明（三线表：符号|含义|单位）
二、模型假设（逐条编号 + 合理性论证）
三、数学模型（目标函数 + 约束 + 公式编号）
四、算法设计（定理+证明+伪代码+复杂度）
五、求解结果（核心表格+解读）
六、敏感性分析
```

**5.2 论文素材包：**
- `key_numbers.md` — 全部关键数字，论文手直接引用
- `参考文献格式.md` — GB/T 7714 格式
- `参考文献精读笔记.md` — 精读论文的可引用句子

**5.3 AI 交互记录模板（每条）：**
```markdown
### 交互 N：[标题]
**所做工作：** [具体做了什么]
**AI 角色：** [AI 生成什么]
**人类角色：** [人类决策]
**采纳情况：** [采纳/修改后采纳/未采纳]
```
关键提示词必须原文照录。

**→ 提示词：** "整理 key_numbers.md，含全部三个问题的核心结果数字。每张图表标注文件名、论文位置、dpi。"

---

### 阶段 6 · 验证与打包（Day 2-3）

**全量验证：** `run_all.bat` 一键跑全部 → CSV 值 vs README 声明 vs HANDOFF 记录，三方交叉验证

**项目整理 checklist：**
- `.gitignore` 含 venv/__pycache__/.pytest_cache
- `requirements.txt` 声明依赖 · `.gitattributes` 统一行尾
- 清理 `__pycache__/` · 修复硬编码路径
- 分步 commit（每阶段一个，别一大坨）

**→ 提示词：** "整理项目：增强 .gitignore、添加 requirements.txt、修复硬编码路径、清理 __pycache__、分步 commit 推送。"

**⚠️ 教训：**
- 代码更新后 model.md/HANDOFF 没同步 → **改代码立即改文档**
- 一次 commit 50 个文件 → **分阶段提交**

---

### 阶段 7 · 论文写作（Day 3 + 缓冲）

```
标题 → 摘要(≤300字) → 关键词(3-5个) → 问题重述 → 问题分析
→ 模型假设(逐条+论证) → 符号说明(三线表) → 模型的建立与求解(每问一节)
→ 结果分析与模型检验 → 模型评价(优点/缺点/改进) → 参考文献 → 附录(代码清单+AI声明)
```

**→ 提示词（论文手用）：** "根据 model.md 和 key_numbers.md，写[某章节]初稿。学术语言、数据准确、公式编号对应。"

## 附录 B：23 次交互核心经验

**可复用（8条）：**
1. 选题用决策矩阵 — 量化而非感觉
2. 先写 model.md 再写代码 — 模型清楚，代码少 bug
3. 多算法交叉对比 — 4 算法 × 4 规模 = 论文讨论有数据
4. 参数全量扫描 — 发现 α≥0.999 是 SA 生效硬门槛
5. 最优性双重下界 — LKH + Held-Karp 严密理论保证
6. 负结果诚实记录 — 3-opt 零改进、空间聚类 +30.7% 都有价值
7. 定理发现是亮点 — 顺序无关性定理、X-排序最优性定理
8. 文献驱动改进 — MVODM、双向取料直接来自精读

**需改进（3条）：**
1. 参数验证太晚 — 应先把赛题参数写进代码常量
2. 文档同步滞后 — 改代码立即改文档
3. 一次性 commit 太大 — 分阶段提交，每阶段一个 commit
